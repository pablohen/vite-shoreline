# vite-shoreline

Simpsons API browser: **Episodes**, **Characters**, and **Locations** tabs with paginated tables, URL-backed search state, and side detail drawers. Built with Vite 8, React 19, TanStack Query + Router, VTEX Shoreline, Zod 4, and Biome. Uses pnpm.

For human-oriented setup and features, see [README.md](README.md).

## Commands

```bash
pnpm install      # install dependencies
pnpm dev          # Vite dev server with HMR
pnpm test         # run unit tests once (Vitest)
pnpm test:watch   # run unit tests in watch mode
pnpm test:coverage # run unit tests with coverage report
pnpm typecheck    # tsc --noEmit
pnpm lint         # biome check .
pnpm format       # biome format --write .
pnpm build        # tsc -b && vite build
pnpm preview      # serve production build locally
```

## Architecture

### URL is the source of truth

Search params validated in `src/routes/__root.tsx`:

- `tab` â€” `episodes` | `characters` | `locations` (default stripped)
- `page` â€” integer â‰Ą 1 (default `1` stripped)
- `detail` â€” optional positive integer; opens drawer for the active tab

Navigation lives in the root route. `SimpsonsBrowsePage` receives callbacks (`onTabChange`, `onPageChange`, `onDetailChange`). Do not add parallel local state for tab, page, or detail.

### Lazy tab fetching

Only the active tab's list query is `enabled` (`src/hooks/use-simpsons-browse-queries.ts`). New tabs must follow the same pattern.

### API layer

All fetches go through `src/api/simpsons-client.ts` with Zod validation in `src/api/simpsons-schemas.ts`. Re-export via `src/simpsons-api.ts`. Do not bypass validation or call the API directly from components.

### Resource config

UI labels, error messages, empty states, and skeleton column counts live in `src/simpsons-resources.ts`. Add copy there, not inline in components.

### Composition for new resources

1. `*Table.tsx` â€” column defs + `SelectableTable`
2. `ResourceBrowseTabPanel` / `ResourceTabPanel` â€” pagination and collection status
3. `*DetailsDrawer.tsx` â€” `DetailsDrawerShell` + `useDetailDrawerContent`
4. Extend `BrowseDrawerState` in `src/hooks/use-browse-drawer-state.ts` with an exhaustive `switch`
5. Add a case to `BrowseDetailDrawer` in `src/components/BrowseDetailDrawer.tsx`

Drawer preview comes from cached list rows; detail query enriches content (`useDetailDrawerContent`).

### Styling

Use Shoreline components and tokens (`@vtex/shoreline`). No Tailwind or CSS modules. Inline `style` is acceptable only for responsive layout tweaks (see `useIsMobileNav` in `src/hooks/use-media-query.ts`).

### Routing

Single file route today (`src/routes/__root.tsx`). Do not hand-edit `src/routeTree.gen.ts`.

## Code style

Project-specific rules (Biome handles the rest via `biome.json`):

- Tabs for indent (width 2)
- Single quotes; semicolons as needed
- Local imports use explicit `.ts` / `.tsx` extensions
- `import type` for type-only imports; all imports at top of file
- PascalCase component files; `use-*` kebab-case hooks; kebab-case utils; `SCREAMING_SNAKE_CASE` constants
- Exhaustive `switch` on tab/resource unions; use `never` in the default case when adding variants
- `as const satisfies` for config maps (see `SIMPSONS_RESOURCE_CONFIG`)

## Boundaries

- **Do not edit:** `src/routeTree.gen.ts`, `dist/`
- **Do not edit unless deps change:** `pnpm-lock.yaml`
- **Do not add:** Redux, Zustand, Tailwind, CSS modules, or a second routing paradigm
- **Do not bypass:** Zod validation on API responses or search params
- **No secrets:** public Simpsons API only; no `.env` required

## Agent workflow

- Keep diffs minimal; match existing patterns before introducing abstractions
- Prefer extending generic shells (`ResourceTabPanel`, `DetailsDrawerShell`, `SelectableTable`) over duplicating layout
- When adding a resource type, touch: schemas, client overloads, tab schema, resource config, table, drawer, browse queries, drawer state switch, and `BrowseDetailDrawer`
- Update this file in the same PR if build commands or architecture change materially
- When committing, use atomic conventional commits (see **Commits** below)

## Commits

Use [Conventional Commits](https://www.conventionalcommits.org/) format:

```
type(optional-scope): imperative subject
```

- **type** â€” one of: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`
- **scope** â€” optional, lowercase; use when it clarifies the area (e.g. `deps`, `tables`, `api`, `drawers`)
- **subject** â€” imperative mood, lowercase, no trailing period; explain *why* when helpful

**Atomicity**

- One logical change per commit â€” a single concern that could be reverted independently
- Split unrelated changes (e.g. feature + docs + lockfile bump â†’ separate commits)
- Do not mix refactors with behavior changes in the same commit
- `chore(deps)` lockfile updates stay in their own commit when possible

**Examples**

```
docs: add AGENTS.md for project overview and architecture
feat(tests): add unit tests for Simpsons API client
refactor(tables): extract shared table cell renderers
chore(deps): upgrade vite to 8.1.0
```

**When to commit**

- Only create commits when the user explicitly asks
